c'est quoi la difference avec les variables des edges fonction alors ?

C'est une très bonne question. Il y a en réalité deux niveaux de variables quand on utilise Supabase Edge Functions auto-hébergé.

1. Variables du conteneur Edge Functions (Coolify)
Ce sont les variables d'environnement du conteneur qui exécute edge-runtime.

Exemple :

SUPABASE_URL=...
SUPABASE_ANON_KEY=...
SUPABASE_SERVICE_ROLE_KEY=...
JWT_SECRET=...
ADMIN_EMAILS=...
Ton code y accède avec :

const emails = Deno.env.get("ADMIN_EMAILS");
Ces variables sont disponibles pour toutes les Edge Functions.

2. Secrets des Edge Functions (supabase secrets)
Avec Supabase (surtout le cloud et aussi le CLI), on peut utiliser :

supabase secrets set ADMIN_EMAILS=...
ou

supabase secrets list
Ces secrets sont injectés automatiquement dans les fonctions lors du déploiement via le CLI.

Sur un projet Supabase Cloud, c'est la méthode recommandée.

Dans ton cas (Supabase auto-hébergé sur Coolify)
Tu n'utilises pas la plateforme Supabase Cloud ni le déploiement via supabase functions deploy.

Tes fonctions sont simplement des dossiers montés dans :

/data/coolify/services/.../volumes/functions
et exécutés directement par edge-runtime.

Par conséquent :

✅ Deno.env.get(...) lit les variables du conteneur.

❌ Les commandes supabase secrets set n'ont généralement aucun effet si tu ne déploies pas les fonctions avec le CLI.

Pourquoi Coolify parle de "Variables des Edge Functions" ?
Si tu as installé Supabase depuis le template Coolify, il existe parfois un écran spécifique pour les Edge Functions. En réalité, ces variables sont simplement injectées dans le conteneur edge-runtime. Elles reviennent au même que des variables d'environnement du conteneur, mais leur portée est limitée à ce service.
